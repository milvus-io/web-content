    </button></h2><p>В типичном конвейере данных AI пользователи могут уже хранить свои данные в формате Parquet или других форматах в своей системе хранения, например AWS S3. Чтобы заставить Milvus использовать эти внешние данные, пользователям обычно нужно импортировать их в собственное хранилище Milvus с помощью конвейеров извлечения-трансформирования-загрузки (ETL).</p>
<p>Такой рабочий процесс "принеси свои данные в Milvus" создает избыточные данные, которые сложно синхронизировать, и увеличивает нагрузку по техническому обслуживанию для обеспечения согласованности данных.</p>
<p>
  
   <span class="img-wrapper"> <img translate="no" src="/docs/v3.0.x/assets/external-collection-bring-data-to-compute.png" alt="Bring data to compute workflow" class="doc-image" id="bring-data-to-compute-workflow" />
   </span> <span class="img-wrapper"> <span>Рабочий процесс "принеси данные на вычисление</span> </span></p>
<p>Чтобы решить эти проблемы, Milvus предоставляет внешние коллекции, которые позволяют вам получать доступ к внешним данным из Milvus, не заботясь о синхронизации данных и ETL-конвейерах.</p>
<p>
  
   <span class="img-wrapper"> <img translate="no" src="/docs/v3.0.x/assets/external-collection-bring-compute-to-data.png" alt="Bring compute to data workflow" class="doc-image" id="bring-compute-to-data-workflow" />
   </span> <span class="img-wrapper"> <span>Перенесите вычисления в рабочий процесс с данными</span> </span></p>
<p>После создания внешняя коллекция может получить прямой доступ к вашим данным и хранить их там же, где вы их храните. В фоновом режиме Milvus создает файлы манифеста для записи сопоставлений между метаданными Milvus и строками во внешних файлах данных. После того как файлы манифеста будут готовы, вы можете создавать индексы во внешней коллекции, как и в любой другой управляемой коллекции.</p>
<p>При изменении данных вручную запускается обновление метаданных за доли секунды, благодаря чему Milvus всегда остается в актуальном состоянии.</p>

      </svg>
    </button></h2><p>Поскольку Milvus не хранит необработанные данные и поддерживает только сопоставление метаданных и необработанных данных, внешние коллекции доступны только для чтения. Это означает, что вы не можете вставлять, поднимать, удалять, импортировать, промывать и уплотнять данные на стороне Milvus.</p>
<p>По сравнению с управляемыми коллекциями внешние коллекции имеют следующие ограничения:</p>
<ul>
<li><p>Вы не можете установить первичный ключ и его атрибуты AutoID.</p></li>
<li><p>Нельзя включить динамическое поле.</p></li>
<li><p>Нельзя задать ключ раздела и ключ кластеризации, поскольку разделы недоступны.</p></li>
<li><p>Вы не можете определить функции в схеме.</p></li>
<li><p>Нельзя изменить схему внешней коллекции после ее создания.</p></li>
<li><p>Нельзя использовать текстовое соответствие с BM25.</p></li>
<li><p>Перед созданием индексов необходимо запустить операцию обновления.</p></li>
</ul>

<p>Операция обновления является асинхронной, поэтому вам нужно настроить итерацию, чтобы следить за ее ходом.</p>
<div class="alert note">
<ul>
<li><p>Операция обновления сканирует метаданные файлов данных и генерирует соответствующие файлы манифеста. Обычно это занимает 150-250 мс.</p></li>
<li><p>В файлах манифеста записывается соответствие между метаданными в Milvus и строками во внешних файлах.</p></li>
<li><p>При обновлении исходных данных необходимо вручную вызвать refresh, чтобы Milvus был в актуальном состоянии.</p></li>
<li><p>Вы не можете индексировать внешнюю коллекцию только после завершения обновления. Однако способ создания индексов такой же, как и для управляемой коллекции.</p></li>
