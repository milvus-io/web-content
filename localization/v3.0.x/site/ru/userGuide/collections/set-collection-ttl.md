      </svg>
    </button></h1><p>Milvus может автоматически исключать сущности с помощью политики <strong>Time-to-Live (TTL)</strong>. Просроченные сущности немедленно перестают отображаться в результатах запросов и поиска и физически удаляются из хранилища при следующем цикле уплотнения - обычно в течение 24 часов.</p>
<p>Существует два режима TTL:</p>
<ul>
<li><p><strong>TTL на уровне коллекции</strong> - одно окно хранения, общее для всех сущностей, задается через свойство <code translate="no">collection.ttl.seconds</code>.</p></li>
<li><p><strong>TTL на уровне сущности</strong> - каждая сущность имеет свое собственное абсолютное время истечения срока хранения в специальном поле <code translate="no">TIMESTAMPTZ</code>, обозначенном как поле TTL через свойство <code translate="no">ttl_field</code>.</p></li>
</ul>
<div class="alert note">
<p>Эта функция применяется только к управляемым коллекциям.</p>
</div>

<li><p>Два режима TTL являются взаимоисключающими. Для коллекции не могут быть одновременно установлены значения <code translate="no">collection.ttl.seconds</code> и <code translate="no">ttl_field</code>. Чтобы переключиться, см. раздел <a href="/docs/ru/set-collection-ttl.md#Migrate-between-the-two-modes">Переход между двумя режимами</a>.</p></li>
<li><p>TTL на уровне коллекции применяет одно окно ко всей коллекции. Если для отдельной строки требуется другое время жизни, используйте TTL на уровне сущности.</p></li>
<li><p>Поле для TTL на уровне сущности должно быть <code translate="no">TIMESTAMPTZ</code>. Другие типы отклоняются.</p></li>
<li><p>Одно поле TTL на коллекцию. Схема может содержать несколько полей <code translate="no">TIMESTAMPTZ</code>, но только одно из них может быть названо <code translate="no">ttl_field</code>.</p></li>
<li><p>Удаление <code translate="no">ttl_field</code> не восстанавливает просроченные сущности. Чтобы восстановить сущность с истекшим сроком действия, добавьте ее с временной меткой <code translate="no">NULL</code> или с будущим сроком действия.</p></li>
</ul>

        ></path>
      </svg>
    </button></h3><p>TTL является правильным инструментом, когда сохранение является <strong>политикой</strong> - вы заранее знаете, что определенные сущности должны со временем исчезнуть, и хотите, чтобы кластер обеспечивал это без написания вами задания cron.</p>
<p>Типичные сценарии:</p>
<ul>
<li><p><strong>Наборы данных с временными окнами.</strong> Храните только последние N дней журналов, метрик, событий или недолговечных кэшей функций.</p></li>
<li><p><strong>Многопользовательские коллекции.</strong> Разные арендаторы имеют разные окна хранения в одной коллекции.</p></li>
<li><p><strong>Политики хранения на каждую запись.</strong> Время жизни каждого документа в конвейерах IoT, хранилищах документов или хранилищах функций MLOps.</p></li>
<li><p><strong>Смешение горячих и холодных данных.</strong> Короткоживущие сущности сосуществуют с долгосрочными в одной коллекции.</p></li>
<li><p><strong>Истечение срока действия в соответствии с требованиями.</strong> Минимизация данных в стиле GDPR, когда каждая запись имеет свою собственную дату "удаления до".</p></li>
<li><p><strong>Истечение срока годности по факту.</strong> Сущность представляет собой запись, которая действительна только до какого-то абсолютного момента (окончание кампании, истечение сессии).</p></li>
</ul>
<div class="alert note">
<p>Сущности с истекшим сроком действия не будут отображаться в результатах поиска или запросов. Однако они могут оставаться в хранилище до последующего уплотнения данных, которое должно быть выполнено в течение следующих 24 часов.</p>
<p>Вы можете контролировать время запуска уплотнения данных, установив конфигурационный элемент <code translate="no">dataCoord.compaction.expiry.tolerance</code> в конфигурационном файле Milvus.</p>
<p>По умолчанию этот элемент конфигурации имеет значение <code translate="no">-1</code>, указывающее на то, что применяется существующий интервал уплотнения данных. Однако если вы измените его значение на целое положительное число, например <code translate="no">12</code>, уплотнение данных будет запускаться через указанное количество часов после истечения срока действия любых сущностей.</p>
</div>

        ></path>
      </svg>
    </button></h3><p>Эти два режима отвечают на разные вопросы хранения:</p>
<ul>
<li><p><strong>TTL на уровне коллекции</strong> применяет единый срок хранения к каждой сущности. Срок действия каждой сущности истекает по адресу <code translate="no">insert_ts + ttl_seconds</code>.</p></li>
<li><p><strong>TTL на уровне сущности</strong> позволяет каждой сущности хранить свое абсолютное время истечения срока действия в поле <code translate="no">TIMESTAMPTZ</code>. Значение <code translate="no">NULL</code> в этом поле означает, что срок действия сущности не истекает.</p></li>
</ul>
<p>Коллекция использует <strong>один</strong> режим одновременно - оба режима являются взаимоисключающими. Переключение между ними - это многоступенчатая операция; см. раздел Переход между двумя режимами.</p>
<p>Используйте эту таблицу, чтобы выбрать режим:</p>
<table>
   <tr>
     <th><p><strong>Если ваша ситуация...</strong></p></th>
     <th><p><strong>Используйте</strong></p></th>
   </tr>
   <tr>
     <td><p>Каждая сущность в коллекции должна следовать одному и тому же окну хранения</p></td>
     <td><p>TTL на уровне коллекции</p></td>
   </tr>
   <tr>
     <td><p>Сохранение - это "с момента вставки хранить N секунд".</p></td>
     <td><p>TTL на уровне коллекции</p></td>
   </tr>
   <tr>
     <td><p>Разные сущности должны иметь разное время жизни в одной коллекции (для арендатора, горячие/холодные, для документа)</p></td>
     <td><p>TTL на уровне сущности</p></td>
   </tr>
   <tr>
     <td><p>Хранение - это абсолютное время по настенным часам (например, 2027-01-01T00:00:00Z)</p></td>
     <td><p>TTL на уровне сущности</p></td>
   </tr>
   <tr>
     <td><p>Хранение определяется временной меткой предприятия, а не временной меткой вставки</p></td>
     <td><p>TTL на уровне сущности</p></td>
   </tr>
   <tr>
     <td><p>Вы хотите обновить или продлить время жизни сущности после вставки</p></td>
     <td><p>TTL на уровне сущности</p></td>
   </tr>
   <tr>
     <td><p>Некоторые сущности не должны истекать, а другие должны</p></td>
     <td><p>TTL на уровне сущности (используйте NULL для бессмертных сущностей)</p></td>
   </tr>
</table>
