<p>Версия Milvus v3.0-beta знаменует начало перехода Milvus от векторной базы данных к семантически-нативному движку озера данных. Ядро Milvus теперь может работать напрямую с данными в открытых форматах озера данных, а основные возможности Milvus были расширены в таких областях, как поиск, схема, жизненный цикл, язык и операции.</p>
<p>Внешний сбор данных и создание моментальных снимков — это основные нововведения в части «озера». Это же ядро лежит в основе Zilliz Lakebase — семантически-нативной платформы данных, построенной на Milvus 3.0.</p>

    </button></h3><h4 id="External-Collection" class="common-anchor-header">External Collection</h4><p>В типичных конвейерах данных для ИИ терабайты вложений и метаданных уже хранятся в объектном хранилище в виде таблиц Parquet, Lance или Iceberg. Копирование этих данных в Milvus удваивает затраты на хранение, добавляет конвейер ETL, который необходимо поддерживать в синхронизации, и лишает клиента контроля над управлением данными.</p>
<p>Функция «Внешний сбор» устраняет необходимость копирования. Коллекция Milvus может ссылаться на файлы там, где они уже находятся, а Milvus управляет только схемой, индексами и выполнением запросов. Инкрементное обновление обеспечивает синхронизацию коллекции с базовыми файлами. Клиенты, чьи данные не могут покидать озеро данных (например, команды из сферы финансов и здравоохранения), могут выполнять векторный поиск по этим данным прямо на месте их хранения. Один набор данных, находящийся в озере данных, также может одновременно обслуживаться несколькими экземплярами Milvus.</p>
<p>Дополнительные сведения см. в разделе <a href="/docs/ru/create-an-external-collection.md">«Создание внешней коллекции</a>».</p>
<h4 id="Snapshot" class="common-anchor-header">Снимок</h4><p>Для обслуживания и пакетного обнаружения часто требуется одновременный доступ к одной и той же коллекции. Оценка A/B-моделей, крупномасштабная дедупликация, проверка заполнения пробелов и откат версий — все эти операции требуют стабильного представления коллекции, даже когда в неё продолжают записываться данные.</p>
<p>Снимки создают доступное только для чтения представление коллекции на определенный момент времени, ссылаясь на существующие сегменты вместо копирования данных, поэтому предельные затраты на хранение близки к нулю. Пакетные задания могут считывать данные из снимка в режиме изоляции типа MVCC, в то время как активная коллекция продолжает принимать записи.</p>
<p>Дополнительную информацию см. в разделах <a href="/docs/ru/snapshots.md">«Снимки»</a>, <a href="/docs/ru/manage-snapshots.md">«Управление снимками</a>» и <a href="/docs/ru/snapshot-use-cases.md">«Варианты использования снимков</a>».</p>
<h4 id="External-Backfill" class="common-anchor-header">Внешнее пополнение</h4><p>Ранее обновление модели вложений, например переход с вложений версии v1 на v2 в существующей коллекции, означало пересоздание с нуля. Это вынуждало либо приостанавливать работу сервиса, либо использовать логику двойной записи на стороне приложения.</p>
<p>Milvus 3.0 поддерживает обновление в режиме «горячего» рабочего процесса. Вы можете добавить новое векторное поле с помощью команды ` <code translate="no">AddCollectionField</code>`, использовать «Снимок» для фиксации согласованной начальной точки, запустить задание встраивания в автономном режиме на основе «Снимка» и записать значения обратно через обычные каналы поступления данных. После того как новое поле будет проиндексировано в режиме онлайн, приложение сможет переключиться на него без простоев.</p>
<h4 id="Query--Search-Order-By" class="common-anchor-header">Запросы / Поиск с сортировкой</h4><p>Поиск и запросы теперь поддерживают сортировку по нескольким полям, при этом сортировка переносится в ядро Milvus, а параметры « <code translate="no">ASC</code> » и « <code translate="no">DESC</code> » можно настраивать для каждого поля отдельно. Это устраняет распространённый недостаток в производственной среде: сортировка Top-K только по расстоянию часто не соответствует бизнес-потребностям, когда наиболее похожий элемент не является самым дешёвым, самым свежим или самым популярным.</p>
<p>Приложениям больше не нужно избыточно извлекать результаты и повторно сортировать их на стороне клиента для формирования составного ранжирования.</p>
<p>Дополнительную информацию см. в разделах <a href="/docs/ru/single-vector-search.md#Sort-Search-Results-by-Scalar-Fields--Milvus-30x">«Сортировка результатов поиска по скалярным полям</a> » и <a href="/docs/ru/get-and-scalar-query.md#Sort-Query-Results--Milvus-30x">«Сортировка результатов запросов</a>».</p>
<h4 id="Null-Vector" class="common-anchor-header">Нулевой вектор</h4><p>Встраивания часто генерируются асинхронно, поэтому сущность может поступить раньше, чем ее вектор. Мультимодальные данные также имеют естественные пробелы, такие как видео без субтитров или товар без изображения. В предыдущих версиях не было подходящего решения: приложения либо откладывали запись до готовности вектора, либо заполняли местозаполняющим вектором, и оба варианта ухудшали качество поиска.</p>
<p>Milvus 3.0 поддерживает значение NULL для векторных полей всех шести типов векторов. Поиск автоматически пропускает векторы с NULL, качество поиска не ухудшается, а векторы с NULL практически не занимают места в хранилище. В рамках этого изменения функция « <code translate="no">AddField</code> » также распространяется на векторные поля: с помощью команды ` <code translate="no">nullable=True</code>` существующая коллекция может добавлять новые векторные поля в режиме онлайн без перестроения.</p>
<p>Дополнительную информацию см. в разделе <a href="/docs/ru/nullable-and-default.md">«Поля, допускающие значение NULL</a>».</p>
<h4 id="Custom-Dictionary--Synonym-Dictionary" class="common-anchor-header">Пользовательский словарь и словарь синонимов</h4><p>Готовые токенизаторы не всегда соответствуют требованиям к качеству поиска в производственной среде. Китайский язык, вертикальные области, такие как медицина, право и химия, а также многоязычные корпуса могут существенно выиграть от использования пользовательских словарей и таблиц синонимов. До сих пор эти ресурсы в основном использовались в виде переформулировок запросов на стороне приложения.</p>
<p>В Milvus 3.0 добавлен механизм FileResource для регистрации пользовательских словарей токенизаторов, списков синонимов, списков стоп-слов и правил декомпозиции сложных слов. После регистрации на ресурс можно ссылаться из любого токенизатора или фильтра, и он будет применяться в BM25, анализаторах и Text Match. Теперь словари и синонимы можно версионировать и централизованно управлять ими, вместо того чтобы они были разбросаны по коду приложения.</p>
<p>Дополнительную информацию см. в разделе <a href="/docs/ru/manage-file-resources.md">«Управление файловыми ресурсами</a>».</p>
<h4 id="Entity-TTL" class="common-anchor-header">Срок хранения сущностей (TTL)</h4><p>TTL на уровне коллекции и на уровне раздела слишком грубы для многих сценариев жизненного цикла и обеспечения соответствия требованиям. Различные арендаторы внутри одной и той же коллекции часто имеют разные правила хранения, и отдельные сущности могут нуждаться в истечении срока хранения по графику, не совпадающему с остальной частью коллекции.</p>
<p>Milvus 3.0 поддерживает TTL на уровне сущностей. Определите поле « <code translate="no">TIMESTAMPTZ</code> » в схеме, отметьте его как поле TTL с помощью свойства коллекции, и Milvus автоматически удалит сущности, срок хранения которых истек. Это позволяет обрабатывать запросы на «право на забвение», удалять данные сеансов по истечении срока и ограничивать историю переписки без необходимости очистки со стороны приложения.</p>
<p>Дополнительную информацию см. в разделе <a href="/docs/ru/set-collection-ttl.md#Set-entity-level-TTL--Milvus-30x">«Установка TTL на уровне сущности</a>».</p>
<h4 id="MinHash-DIDO-Doc-in-Doc-out" class="common-anchor-header">MinHash DIDO (Doc-in, Doc-out)</h4><p>В Milvus 2.6 был добавлен индекс <code translate="no">MINHASH_LSH</code> для обнаружения почти-дубликатов на основе наборов, но приложениям по-прежнему приходилось вычислять сигнатуры MinHash перед записью данных в Milvus.</p>
<p>В Milvus 3.0 добавлена функция MinHash, выполняемая на стороне сервера. Объявите в схеме поле ввода « <code translate="no">VARCHAR</code> » и поле вывода « <code translate="no">BINARY_VECTOR</code> », привяжите функцию « <code translate="no">FunctionType.MINHASH</code> », и Milvus будет вычислять сигнатуры во время вставки, массовой вставки и поиска. В сочетании с функцией « <code translate="no">MINHASH_LSH</code> » это обеспечивает поддержку рабочих процессов дедупликации для больших наборов данных, формирования отпечатков и обнаружения плагиата внутри Milvus.</p>
<p>Дополнительную информацию см. в разделе <a href="/docs/ru/minhash-function.md">«Функция MinHash</a>».</p>
<h4 id="EmbList-+-DISKANN" class="common-anchor-header">EmbList + DISKANN</h4><p>Предположение «одна сущность = один вектор» больше не соответствует современному поиску. Длинные документы разбиваются на множество фрагментов, модели с поздним взаимодействием, такие как ColBERT, генерируют по одному вектору на каждый токен, а мультимодальные сущности могут иметь несколько представлений.</p>
<p>EmbList хранит список векторов переменной длины для каждого объекта, используя « <code translate="no">DISKANN</code> » в качестве индекса на диске. Использование дискового пути позволяет контролировать загрузку ОЗУ, когда объем корпуса превышает доступный объем памяти. EmbList + « <code translate="no">DISKANN</code> » — это первый вариант более широкого семейства StructList в данной версии RC. Остальные элементы семейства, включая фильтрацию StructList и ускорение работы с несколькими векторами с помощью Muvera / Lemur, планируется включить в официальный релиз версии 3.0.</p>
<p>Дополнительную информацию см. в разделе <a href="/docs/ru/search-with-embedding-lists.md">«Поиск с использованием списков вложений</a>».</p>
<h4 id="Force-Merge" class="common-anchor-header">Принудительное слияние</h4><p>В производственных рабочих нагрузках со временем накапливается фрагментация сегментов, что приводит к колебаниям задержки запросов и увеличению объема хранилища.</p>
<p>В Milvus 3.0 добавлена возможность явного запуска уплотнения сегментов в периоды низкой нагрузки как в синхронном, так и в асинхронном режимах.</p>
<p>Дополнительную информацию см. в разделе <a href="/docs/ru/force-merge.md">«Принудительная уплотнение слиянием</a>».</p>
