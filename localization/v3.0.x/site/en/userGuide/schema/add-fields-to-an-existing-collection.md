    </button></h1><p>As a collection moves from development to production, the fields around each entity often change. You might add scalar fields such as <code translate="no">source_uri</code> or <code translate="no">review_status</code> for filtering and application logic, add a new vector field for embeddings generated by your application, add a BM25-generated sparse vector field for lexical search over existing text, or remove fields that are no longer used. Alter Collection Schema lets you make supported field changes in place instead of recreating the collection.</p>
<div class="alert note">
<p>This guide covers field-level schema changes in managed collections, including user-defined fields and vector fields generated by functions. To add a field to an external collection, refer to <a href="/docs/alter-external-collection-schema.md">Alter External Collection Schema</a>. For field property changes, such as changing <code translate="no">max_length</code> on a <code translate="no">VARCHAR</code> field or <code translate="no">max_capacity</code> on an <code translate="no">ARRAY</code> field, refer to <a href="/docs/alter-collection-field.md">Alter Collection Field</a>. For dynamic field behavior, refer to <a href="/docs/enable-dynamic-field.md">Dynamic Field</a> and <a href="/docs/modify-collection.md">Modify Collection</a>.</p>
</div>

      </svg>
    </button></h2><p><strong>Add user-defined fields</strong></p>
<ul>
<li><p>Added user-defined fields must be nullable. Set <code translate="no">nullable=True</code> when calling <code translate="no">add_collection_field()</code>. For existing entities, the added field is <code translate="no">NULL</code> unless you add a scalar field with a <code translate="no">default_value</code>.</p></li>
<li><p>Adding user-defined scalar fields is supported in Milvus 2.6.x and later. Adding user-defined vector fields is supported in Milvus 2.6.18 and later.</p></li>
<li><p>Field names must be unique among fields in the collection.</p></li>
</ul>
<p><strong>Add vector fields generated by functions</strong></p>
<ul>
<li><p>Each schema update can add only one function and one generated vector field.</p></li>
<li><p>The supported function determines the generated vector field type: <code translate="no">BM25</code> generates a <code translate="no">SPARSE_FLOAT_VECTOR</code> field, and <code translate="no">MINHASH</code> generates a <code translate="no">BINARY_VECTOR</code> field.</p></li>
<li><p>The generated vector field must be a new field. It cannot point to a field that already exists in the collection schema.</p></li>
<li><p>The generated vector field cannot be nullable.</p></li>
<li><p>The input fields used by the function must already exist in the collection.</p></li>
</ul>
<p><strong>Drop user-defined fields</strong></p>
<ul>
<li><p>You cannot drop the primary key field, partition key field, clustering key field, or the last vector field in a collection.</p></li>
<li><p>You can drop a whole <code translate="no">ARRAY&lt;STRUCT&gt;</code> field, but you cannot drop an individual sub-field inside an <code translate="no">ARRAY&lt;STRUCT&gt;</code> field.</p></li>
<li><p>You cannot directly drop a field that is used as a function input field or generated as a function output field. To remove a function output field, drop the function that generates it.</p></li>
</ul>
<p><strong>Drop vector fields generated by functions</strong></p>
<ul>
<li><p>In this schema-change workflow, dropping a function removes the function and its generated output fields. Function input fields remain in the collection schema.</p></li>
<li><p>Dropping a function is rejected if removing its output fields would leave the collection without any vector field.</p></li>
</ul>
<div class="alert note">
<p>For schema changes outside supported add and drop operations, recreate or migrate the collection.</p>
</div>

      </svg>
    </button></h3><p>Use <code translate="no">add_collection_field()</code> to add a user-defined scalar field to an existing collection.</p>
<p>This differs from storing arbitrary keys in the dynamic field: after the schema update is available, the new scalar field becomes a regular part of the collection schema. You can insert or upsert values into it, create indexes on it where supported, use it in queries and search filters, and return it in query or search output.</p>
<p>Because existing entities were inserted before the new field existed, every added user-defined scalar field must be nullable:</p>
<ul>
<li><p>If you add a scalar field with <code translate="no">nullable=True</code> and no <code translate="no">default_value</code>, existing entities return <code translate="no">NULL</code> for the new field.</p></li>
<li><p>If you add a scalar field with <code translate="no">nullable=True</code> and <code translate="no">default_value</code>, existing entities return the default value instead of <code translate="no">NULL</code>.</p></li>
</ul>
<p>Scalar filter expressions do not match <code translate="no">NULL</code> scalar values. For details, refer to <a href="/docs/nullable-and-default.md">Nullable Fields</a>.</p>

<p>The following example adds a nullable <code translate="no">source</code> field to an existing collection named <code translate="no">product_catalog</code>.</p>

<span class="highlighted-comment-line">    field_name=<span class="hljs-string">&quot;source&quot;</span>,</span>
<span class="highlighted-comment-line">    data_type=DataType.VARCHAR,</span>
<span class="highlighted-comment-line">    max_length=<span class="hljs-number">128</span>,</span>
<span class="highlighted-comment-line">    nullable=<span class="hljs-literal">True</span>,</span>
<span class="highlighted-comment-line">)</span>
<button class="copy-code-btn"></button></code></pre>
<p>After the field is added, entities that already existed in the collection return <code translate="no">NULL</code> for <code translate="no">source</code>. New entities can set <code translate="no">source</code> during insert or upsert.</p>

<p>If existing entities should return a concrete value instead of <code translate="no">NULL</code>, specify <code translate="no">default_value</code> when adding a scalar field. <p>Adding a vector field generated by a function adds two related parts to the collection schema:</p>
<ul>
<li><p>A generated vector output field, such as <code translate="no">text_sparse</code>, that stores values generated by Milvus.</p></li>
<li><p>A function, such as <code translate="no">text_bm25</code>, that reads from existing input fields and writes to the generated vector field.</p></li>
</ul>
<p>The supported function determines the generated vector field type:</p>
<table>
<thead>
<tr><th>Function</th><th>Generated vector field type</th><th>Typical input field</th><th>Typical use case</th></tr>
</thead>
<tbody>
<tr><td><code translate="no">BM25</code></td><td><code translate="no">SPARSE_FLOAT_VECTOR</code></td><td>A <code translate="no">VARCHAR</code> or <code translate="no">TEXT</code> field with analyzer enabled</td><td>Lexical search and keyword relevance</td></tr>
<tr><td><code translate="no">MINHASH</code></td><td><code translate="no">BINARY_VECTOR</code></td><td>A <code translate="no">VARCHAR</code> or <code translate="no">TEXT</code> field</td><td>Near-duplicate detection</td></tr>
</tbody>
</table>
<p>For details about how each function works, refer to <a href="/docs/bm25-function.md">BM25 Function</a> and <a href="/docs/minhash-function.md">MinHash Function</a>.</p>
<p>The generated vector field must not already exist in the collection, and it cannot be nullable. The function input field must already exist.</p>
<p><strong>Example: Add a BM25-generated sparse vector field for lexical search</strong></p>
<p>The following example adds a BM25 function named <code translate="no">text_bm25</code> and a generated sparse vector field named <code translate="no">text_sparse</code> to an existing collection. The collection must already have a <code translate="no">VARCHAR</code> or <code translate="no">TEXT</code> field named <code translate="no">text</code> with analyzer enabled.</p>
<pre><code translate="no" class="language-python"><span class="hljs-keyword">from</span> pymilvus <span class="hljs-keyword">import</span> DataType, Function, FunctionType, MilvusClient

)

    output_field_names=[<span class="hljs-string">&quot;text_sparse&quot;</span>],

)

<span class="highlighted-comment-line">client.add_function_field(</span>
<span class="highlighted-comment-line">    collection_name=<span class="hljs-string">&quot;product_catalog&quot;</span>,</span>
<span class="highlighted-comment-line">    field_schema=sparse_field,</span>
<span class="highlighted-comment-line">    func=bm25_function,</span>
<span class="highlighted-comment-line">)</span>

<p>Conceptually, this operation adds the following field and function definitions:</p>
<pre><code translate="no" class="language-plaintext">New generated output field:

For the complete BM25 search workflow, refer to <a href="/docs/full-text-search.md">Full Text Search</a>.</p>
<p>Milvus also supports MinHash-generated binary vector fields for near-duplicate detection. A MinHash function uses <code translate="no">FunctionType.MINHASH</code> and writes to a new <code translate="no">BINARY_VECTOR</code> output field.
