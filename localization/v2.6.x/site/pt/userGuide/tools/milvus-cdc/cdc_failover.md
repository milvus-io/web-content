<p>Este guia pressupõe que a topologia original é:</p>
<pre><code translate="no" class="language-text">cluster-a (primary)  -&gt;  cluster-b (standby)

<p>Após o failover, <code translate="no">cluster-b</code> torna-se um primário autónomo:</p>
<pre><code translate="no" class="language-text">cluster-b (primary)
<button class="copy-code-btn"></button></code></pre>

    </button></h2><p>Use o failover somente quando:</p>
<ul>
<li>O primário original não puder responder às solicitações.</li>
<li>O primário não pode ser recuperado dentro de um tempo aceitável.</li>
<li>A restauração da disponibilidade de gravação é mais importante do que esperar pelo primário antigo.</li>
</ul>
<p>Se o primário ainda estiver acessível, use <a href="/docs/pt/v2.6.x/cdc_switchover.md">o Switchover</a>. O Switchover evita a perda de dados.</p>

<p>A possível perda de dados é determinada pelo atraso do CDC no momento em que o primário ficou indisponível.</p>
<p>Antes de executar o failover, entenda a compensação:</p>

    </button></h2><p>Confirme o seguinte:</p>
<ul>
<li>O primário original não está disponível.</li>
<li>Você decidiu não esperar pela recuperação do primário.</li>
<li>O tráfego de aplicações pode ser redireccionado para o standby.</li>
<li>A automação de tráfego não enviará gravações de volta para o primário antigo se ele se recuperar.</li>
<li>Você tem o ID do cluster em espera, o endereço, o token e os canais p.</li>
</ul>
<p>O requisito de segurança mais importante é evitar o split brain. Após o failover, apenas o standby promovido deve aceitar gravações de aplicativos.</p>

<li>Manter os clusters em espera provisionados para lidar com a taxa de gravação primária.</li>

<li>Repetir as gravações cujo sucesso é incerto após o failover.</li>
<li>Preferir a transição sempre que o primário ainda puder responder.</li>
</ul>
